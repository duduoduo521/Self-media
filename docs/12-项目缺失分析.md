## 一、严重 Bug（必须立即修复）

### 1.1 登出未生效

| 项 | 详情 |
|---|---|
| **位置** | `crates/self-media-web/src/auth.rs:58-63` |
| **现象** | `logout` 处理器仅返回 `Ok(ApiOk(()))`，未调用 `user_service.logout()` 删除服务端 session |
| **影响** | 用户点击登出后 session 仍然有效，JWT 在过期前可继续使用 |
| **修复方案** | 从 `AuthUser` 提取 token，调用 `user_service.logout(token)` 删除数据库 session 记录 |

```rust
// 当前代码（错误）
async fn logout(
    auth: AuthUser,
    State(_state): State<AppState>,
) -> Result<ApiOk<()>, WebError> {
    Ok(ApiOk(()))
}

// 修复后
async fn logout(
    auth: AuthUser,
    State(state): State<AppState>,
) -> Result<ApiOk<()>, WebError> {
    state.user_service.logout(auth.user_id, &auth.token).await?;
    Ok(ApiOk(()))
}
```

> **前置**：`AuthUser` 结构体需增加 `token` 字段，认证中间件需在提取 `user_id` 时同时保存原始 token。

### 1.2 API Key 设置必失败

| 项 | 详情 |
|---|---|
| **位置** | `crates/self-media-web/src/config.rs:32-35` |
| **现象** | `set_api_key` 调用 `user_key_cache.get_or_derive(user_id, &db, None)`，第三个参数 `password=None` |
| **影响** | 缓存未命中时，`get_or_derive` 尝试从数据库加载但无法解密用户密钥（缺少密码），返回 "用户密钥未缓存，需重新登录" 错误 |
| **根因** | 用户登录后密码不持久化在前端，Web 端无法在后续请求中提供密码用于派生用户密钥 |
| **修复方案** | 登录成功后，将用户密钥缓存到 `UserKeyCache`，后续请求直接从缓存获取 |

```rust
// 方案：在 login 处理器中，登录成功后缓存用户密钥
async fn login(
    State(state): State<AppState>,
    Json(body): Json<LoginRequest>,
) -> Result<ApiOk<Session>, WebError> {
    let (session, user_key) = state.user_service.login(&body.username, &body.password).await?;
    // 登录时缓存用户密钥
    state.user_key_cache.insert(session.user_id, user_key).await;
    Ok(ApiOk(session))
}
```

> **前置**：`UserService::login` 需返回 `(Session, UserKey)` 而非仅 `Session`；`UserKeyCache` 需增加 `insert` 方法。

### 1.3 系统密钥未持久化

| 项 | 详情 |
|---|---|
| **位置** | `crates/self-media-web/src/main.rs:179` |
| **现象** | 每次 `SystemKey::generate()` 生成新密钥，服务器重启后所有已签发的 JWT 失效 |
| **影响** | 用户需要重新登录；运行中的任务状态丢失 |
| **修复方案** | 使用 `.env` 中的 `SELF_MEDIA_MACHINE_KEY` 加密持久化系统密钥到文件，启动时尝试加载 |

```
启动流程：
1. 检查密钥文件（如 ./data/system_key.enc）是否存在
2. 存在 → 用 MACHINE_KEY 解密加载
3. 不存在 → 生成新密钥，用 MACHINE_KEY 加密后写入文件
```

### 1.4 TaskStatus 序列化错误

| 项 | 详情 |
|---|---|
| **位置** | `crates/self-media-core/src/task/scheduler.rs:149` |
| **现象** | `serde_json::to_string(&status)` 将枚举序列化为 `"\"Running\""`（带引号的 JSON 字符串），而数据库 `DEFAULT 'Pending'` 期望的值为 `"Running"` |
| **影响** | 任务状态更新后数据库存储值格式错误，查询匹配失败 |
| **修复方案** | 改用 `serde_json::to_value(status).as_str()` 或确保枚举使用 `#[serde(rename_all = "PascalCase")]` 后直接 `to_string` |

```rust
// 当前（错误）：serde_json::to_string 对枚举产生双重引号
let status_str = serde_json::to_string(status)?;  // 结果: "\"Running\""

// 修复：提取枚举的字符串值
let status_str = serde_json::to_value(status)?
    .as_str()
    .ok_or_else(|| AppError::internal("状态序列化失败"))?
    .to_string();  // 结果: "Running"
```

## 三、安全缺陷

### 3.1 Cookie 安全属性缺失

| 项 | 详情 |
|---|---|
| **现象** | JWT 通过 Cookie 传递，但未设置 `HttpOnly` / `Secure` / `SameSite` 属性 |
| **影响** | XSS 可窃取 token；中间人可截获；CSRF 可冒充请求 |
| **设计文档要求** | `Set-Cookie: token=<jwt>; HttpOnly; Secure; SameSite=Strict; Path=/; Max-Age=604800` |
| **当前实现** | 登录响应仅返回 JSON body 中的 session，未设置 Cookie |

### 3.2 前端 Token 注入缺失

| 项 | 详情 |
|---|---|
| **位置** | `frontend/src/api/http.ts` |
| **现象** | axios 配置了 `withCredentials: true`，但登录后未在请求头附加 `Authorization: Bearer xxx` |
| **影响** | 后端认证中间件从 Cookie 提取 token，但前端登录响应的 token 存储在 Pinia store 中，从未写入 Cookie |
| **修复方案** | 登录成功后将 token 写入 Cookie，或在 axios 请求拦截器中添加 `Authorization` header |

### 3.3 CSRF 未启用

| 项 | 详情 |
|---|---|
| **现象** | `axum-csrf` 依赖已引入但未在 `main.rs` 中使用 |
| **影响** | Web 端所有状态变更请求（POST/PUT/DELETE）可被 CSRF 攻击 |
| **设计文档** | `docs/10-API接口设计.md` 定义了 CSRF 防护层，但未实现 |

### 3.4 任务/配置接口缺少权限校验

| 项 | 详情 |
|---|---|
| **位置** | `task.rs:59-66`, `task.rs:68-76` |
| **现象** | `get_task` / `cancel_task` 的 `_auth: AuthUser` 未验证 task 是否属于当前用户 |
| **影响** | 任何已登录用户可查看/取消其他用户的任务 |
