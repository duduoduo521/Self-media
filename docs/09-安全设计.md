# 安全设计详细文档

---

## 1. 凭证存储安全

| 数据类型 | 存储方式 | 加密算法 | 密钥来源 |
|---------|---------|---------|---------|
| 用户密码 | Argon2id 哈希 + 随机 Salt | Argon2id (mem=64MB, time=3, lanes=4) | 无需密钥，单向哈希 |
| MiniMax API Key | AES-256-GCM 加密存储 | AES-256-GCM + 随机 Nonce | UserKey（用户密码派生） |
| 平台 Cookie | AES-256-GCM 加密存储 | AES-256-GCM + 随机 Nonce | UserKey（用户密码派生） |
| 会话 Token | JWT (HS256) | HMAC-SHA256 | SystemKey（随机生成） |
| 系统密钥本身 | AES-256-GCM 加密存储 | AES-256-GCM + 随机 Nonce | 机器密钥 |

## 2. 密钥派生链

### 2.1 系统密钥链

```
机器特征 / 环境变量
        │
        ▼  SHA-256 / 直接使用
   机器密钥 (32 bytes)
        │
        ▼  AES-256-GCM 加密
   系统密钥 (32 bytes, 随机生成)
        │
        └──► JWT (HS256) 签名/验证
```

### 2.2 用户密钥链

```
用户密码 + 随机 Salt (16 bytes)
        │
        ▼  Argon2id
   用户密钥 (32 bytes)
        │
        ├──► 加密 API Key
        ├──► 加密 Cookie
        └──► 解密 API Key / Cookie

   （不用于 JWT 签名 — 解决了旧方案的验证死循环）
```

### 2.3 密码修改重加密流程

```
旧密码 ──► 旧用户密钥 ──► 解密所有敏感数据
                                │
新密码 ──► 新用户密钥 ◄─────────┘
                │
                └──► 重新加密所有敏感数据
                     更新 password_hash + salt
                     清除所有会话（强制重新登录）
```

## 3. 安全约束

### 3.1 密钥管理

- 系统密钥仅在内存中存在明文，持久化时使用机器密钥加密
- 用户密钥仅在内存中存在，不持久化
- 用户密钥实现 `Drop` trait，离开作用域时自动零化内存
- 应用退出后用户密钥销毁，下次登录重新派生
- Web 端用户密钥通过 LRU 缓存管理，设置容量上限

### 3.2 数据存储

- SQLite 数据库文件权限设为 0600（仅当前用户可读写）
- SQLite 启用 WAL 模式
- 加密数据格式：`Base64(Nonce[12] + Ciphertext + Tag[16])`

### 3.3 日志脱敏

```rust
/// 日志脱敏规则
pub struct LogSanitizer;

impl LogSanitizer {
    /// 脱敏 API Key：保留前 4 位，其余用 * 替代
    pub fn mask_api_key(key: &str) -> String {
        if key.len() <= 4 {
            return "*".repeat(key.len());
        }
        format!("{}{}", &key[..4], "*".repeat(key.len() - 4))
    }

    /// 脱敏 Cookie：不输出
    pub fn mask_cookie(_: &str) -> String {
        "[COOKIE_REDACTED]".to_string()
    }

    /// 脱敏 Token：保留前 8 位
    pub fn mask_token(token: &str) -> String {
        if token.len() <= 8 {
            return "*".repeat(token.len());
        }
        format!("{}{}", &token[..8], "*".repeat(token.len() - 8))
    }
}
```

禁止在日志中输出：
- 完整 API Key
- 完整 Cookie
- 完整 JWT Token
- 完整加密密钥
- 用户密码（含哈希前的明文）

### 3.4 Web 端安全

- 所有敏感接口强制 HTTPS
- 启用 CSRF Token 中间件（`axum-csrf`）
- Cookie 设置 `HttpOnly`、`Secure`、`SameSite=Strict`
- AI 生成内容渲染前通过 DOMPurify sanitize（防 XSS）

### 3.5 密码安全

- 使用 `subtle::ConstantTimeEq` 进行密码哈希比较（防时序攻击）
- 密码强度要求：长度 8-64，必须包含大小写字母 + 数字
- 登录失败不透露具体原因（统一返回"用户名或密码错误"）

## 4. 输入校验安全

| 输入项 | 校验规则 | 防护目标 |
|--------|---------|---------|
| 用户名 | 长度 3-32，仅字母/数字/下划线 | SQL 注入（配合参数化）、XSS |
| 密码 | 长度 8-64，大小写+数字 | 暴力破解 |
| 主题/内容 | 长度 ≤ 500 字 | 缓冲区溢出、API 滥用 |
| API Key | 非空，格式校验 | 空值注入 |
| 平台 Cookie | 非空，格式校验 | 空值注入 |
| 所有文本输入 | trim 后校验 | 空白字符绕过 |

> sqlx 参数化查询已防护 SQL 注入，业务层校验主要防止逻辑漏洞和资源滥用。
