## 一、严重安全问题（Critical）

### S-01 用户密钥派生算法与安全设计不符

**文件**：`crates/self-media-crypto/src/user_key.rs`

安全设计文档规定使用 **Argon2id（mem=64MB, time=3, lanes=4）** 派生用户密钥，但实际实现是一个自定义的 `pbkdf2_simple` 函数：

```rust
fn pbkdf2_simple(password: &[u8], salt: &[u8], output: &mut [u8]) {
    // SHA-256 + 1000 次迭代
}
```

- 仅 1000 次 SHA-256 迭代，在现代 GPU 上可在毫秒级暴力破解
- 自定义密码哈希算法本身是高风险实践
- 函数名暗示是 PBKDF2 但实现是自制轮换 XOR，并非标准 PBKDF2

### S-02 MiniMax API Key 明文存储（已绕过加密） ✅ 已修复

**文件**：`crates/self-media-core/src/config/service.rs`

`set_api_key` 接收 `_user_key: &UserKey` 参数但完全忽略，将 API Key 明文写入数据库 `users.minimax_api_key` 列：

```rust
pub async fn set_api_key(&self, user_id: i64, _provider: &str, key: &str,
    _region: &str, _user_key: &UserKey) -> Result<(), AppError> {
    sqlx::query("UPDATE users SET minimax_api_key = ? WHERE id = ?")
        .bind(key)            // ← 明文存储，UserKey 被忽略
        ...
}
```

`get_api_key` 同样忽略 UserKey 参数，直接读取明文。安全设计文档明确要求"AES-256-GCM 加密存储"。

**修复方案**：
- `ConfigService` 新增 `system_key: SystemKey` 字段
- `set_api_key` 使用 `system_key.encrypt(key)` 加密后存储
- `get_api_key` 使用 `system_key.decrypt(encrypted_key)` 解密后返回

### S-06 默认机器密钥长度错误，导致系统密钥每次重启后失效 ✅ 已修复

**文件**：`crates/self-media-web/src/main.rs`、`crates/self-media-crypto/src/system_key.rs`

```rust
let machine_key = std::env::var("SELF_MEDIA_MACHINE_KEY")
    .unwrap_or_else(|_| "dev-machine-key-change-in-production".to_string());
```

`"dev-machine-key-change-in-production"` 为 **36 字节**，而 AES-256-GCM 要求精确 **32 字节**。

后果：
1. `SystemKey::save()` 调用 `Aes256Gcm::new_from_slice(machine_key_bytes)` 时失败（返回"机器密钥长度错误"）
2. `save_system_key` 记录 warn 后跳过，密钥文件写入失败
3. 每次服务重启都会重新生成一个新的系统密钥
4. 所有已签发的 JWT token 在重启后全部失效，用户被强制登出
5. 若用户未设置 `SELF_MEDIA_MACHINE_KEY`，则根本无法持久化系统密钥

**修复方案**：
- `system_key.rs` 添加 `MACHINE_KEY_MIN_LEN = 16` 常量
- 默认机器密钥改为 `"dev-machine-key-32bytes!!"`（精确32字节）
- `load()` 函数添加长度验证和截断/填充处理

### Q-01 密码强度校验与安全文档不符 ✅ 已修复

**文件**：`crates/self-media-core/src/user/service.rs`
