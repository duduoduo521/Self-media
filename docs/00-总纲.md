# 自媒体一键自动发布系统 — 总纲

> 版本：v1.1 | 日期：2026-04-09 | 状态：设计阶段

---

## 一、项目概述

### 1.1 产品定位

面向国内自媒体创作者的一站式 AI 内容生成与多平台自动发布桌面/Web 应用。核心价值：**热点发现 → AI 生成 → 自动配图/配音 → 一键多平台发布**，全链路零人工干预。

### 1.2 核心需求

| 编号 | 需求 | 优先级 |
|------|------|--------|
| R1 | MiniMax AI 深度集成（文本/图片/视频/语音生成） | P0 |
| R2 | 国内 6 大自媒体平台自动发布（小红书/抖音/公众号/B站/微博/头条） | P0 |
| R3 | 用户注册 + 本地加密登录 | P0 |
| R4 | 一键生成：文本模式（文案+自动配图） / 视频模式（脚本+视频+配音） | P0 |
| R5 | 热点检索与选择 | P1 |
| R6 | 跨平台：Windows / macOS / Web | P0 |

### 1.3 与参考方案的差异说明

| 维度 | 参考方案 | 本方案 | 理由 |
|------|---------|--------|------|
| MiniMax 集成 | 捆绑 CLI 二进制，子进程调用 | **直接调用 MiniMax REST API** | CLI 是 Node.js 工具，捆绑需附带 Node 运行时（>50MB）；API 直调更轻量、更可控、更易错误处理 |
| 发布能力 | 简单函数骨架 | **适配器模式 + 平台抽象层 + 签名模块** | 各平台登录/上传/发布机制差异大，需独立适配器隔离变化；各平台均有反爬签名，需独立签名模块 |
| Web 支持 | Tauri 统一 | **Tauri（桌面）+ Axum（Web）共享核心库** | Tauri 不原生支持 Web 部署；Web 端需独立 HTTP 服务 |
| 数据库 | rusqlite | **sqlx + SQLite** | sqlx 异步原生、编译期 SQL 检查、与 tokio 生态无缝集成 |
| 任务调度 | 无设计 | **异步任务队列 + 状态机** | 视频/发布等长任务需进度追踪、取消、重试 |
| 密钥管理 | 单一主密钥 | **系统密钥 + 用户密钥双层架构** | 解决 JWT 跨会话验证与多用户密钥隔离问题 |

### 2.2 核心设计原则

1. **核心库共享**：所有业务逻辑在 `self-media-core` crate 中，Tauri 和 Axum 均依赖此 crate
2. **适配器模式**：平台发布能力通过 trait 抽象，每个平台独立实现，互不影响
3. **异步优先**：全链路 async/await，基于 tokio 运行时
4. **错误传播**：统一 `AppError` 枚举，thiserror 派生，携带错误码，前端友好的错误码
5. **配置外置**：所有密钥、平台配置通过环境变量或加密配置文件管理
6. **密钥双层**：系统密钥（JWT 签名）与用户密钥（数据加密）分离，解决多用户与跨会话问题

## 三、技术选型

### 3.1 后端（Rust）

| 分类 | 选型 | 版本 | 理由 |
|------|------|------|------|
| 异步运行时 | tokio | 1.x | Rust 生态标准，全功能异步 |
| HTTP 客户端 | reqwest | 0.12+ | 支持 multipart、cookie、流式 |
| 数据库 | sqlx + SQLite | 0.8+ | 异步原生、编译期 SQL 检查 |
| 加密 | aes-gcm + argon2 | 最新 | AES-256-GCM 加密存储，Argon2id 密钥派生 |
| 恒定时间比较 | subtle | 4.x | 防时序攻击 |
| JWT | jsonwebtoken | 9.x | JWT 签发与验证 |
| 序列化 | serde + serde_json | 1.x | Rust 标准序列化框架 |
| 错误处理 | thiserror + anyhow | 1.x | 库用 thiserror，应用用 anyhow |
| 日志 | tracing + tracing-subscriber | 0.1.x | 结构化日志，异步友好 |
| 桌面框架 | tauri | 2.x | 跨平台桌面应用 |
| Web 框架 | axum | 0.8+ | 轻量高性能，tokio 生态 |
| CSRF 防护 | axum-csrf | 最新 | Web 端 CSRF Token |
| 模板/构建 | cargo workspace | - | monorepo 管理 |

### 3.2 前端（Vue 3）

| 分类 | 选型 | 理由 |
|------|------|------|
| 框架 | Vue 3 + Composition API | 轻量、响应式、生态成熟 |
| 构建工具 | Vite | 极速 HMR，Tauri 官方推荐 |
| UI 组件库 | Naive UI | 轻量、TypeScript 原生、主题定制 |
| 状态管理 | Pinia | Vue 3 官方推荐 |
| HTTP 客户端 | @tauri-apps/api (桌面) / axios (Web) | 双端适配 |
| XSS 防护 | DOMPurify | 对 AI 生成内容做 sanitize |
| 图标 | @vicons/ionicons5 | 轻量图标库 |

## 六、全局错误码规范

所有 `AppError` 必须携带错误码，前后端通过错误码对接：

| 错误码 | 含义 | 前端处理 |
|--------|------|---------|
| AUTH_001 | 用户名已存在 | 提示换用户名 |
| AUTH_002 | 密码错误 | 提示重新输入 |
| AUTH_003 | 会话过期 | 跳转登录页 |
| AUTH_004 | 用户名格式不合法 | 提示格式要求 |
| AUTH_005 | 密码强度不足 | 提示密码要求 |
| AI_001 | API Key 无效 | 跳转设置页 |
| AI_002 | 生成超时 | 提示重试 |
| AI_003 | 配额不足 | 提示充值 |
| PLAT_001 | Cookie 过期 | 提示重新登录平台 |
| PLAT_002 | 上传失败 | 自动重试 1 次 |
| PLAT_003 | 发布限流 | 延迟重试 |
| PLAT_004 | 签名验证失败 | 提示平台 API 可能变更 |
| TASK_001 | 任务不存在 | 刷新任务列表 |
| TASK_002 | 任务已取消 | 无需处理 |
| TASK_003 | 并发任务数超限 | 提示等待 |
| CONFIG_001 | 配置项不存在 | 使用默认值 |
| CRYPTO_001 | 加解密失败 | 提示重新登录 |
| INPUT_001 | 输入参数校验失败 | 提示具体校验规则 |

## 七、风险与应对

| 风险 | 等级 | 应对策略 |
|------|------|---------|
| 平台 API 变更导致发布失败 | 高 | 适配器模式隔离变化；签名模块独立维护；监控 API 变更；社区及时更新 |
| 平台反爬签名机制升级 | 高 | 签名模块独立 crate，热更新友好；引入 headless browser 作为降级方案 |
| Cookie 过期频繁 | 高 | 登录态健康检查（定时检测）；自动提醒重新登录；研究 refresh 机制 |
| 平台用户协议违规风险 | 高 | 仅辅助创作者提高效率，不用于批量垃圾发布；提示用户遵守平台规则；建议控制发布频率 |
| MiniMax API 配额/限流 | 中 | 请求队列 + 指数退避重试；配额余量预警 |
| 视频生成耗时长（>5min） | 中 | 异步任务 + 进度推送；超时自动取消（最大 30 分钟） |
| 反爬机制升级 | 中 | 请求频率控制；User-Agent 轮换；浏览器指纹模拟 |
| SQLite 并发写入冲突 | 低 | WAL 模式 + 写入串行化；忙等待超时 5s |
| 跨平台打包兼容性 | 低 | CI 多平台构建；平台特定代码条件编译 |

## 八、开发关键注意事项

### 8.1 实现优先级

```
self-media-crypto → self-media-db → self-media-core (共享类型+用户模块)
→ self-media-ai → self-media-publish → self-media-app / self-media-web
```

以上是依赖链，必须严格按序搭建。crypto 和 db 是所有业务模块的基础，共享类型（`Platform`、`HotspotSource`、`AppError`、`ApiError` 等）定义在 `self-media-core/src/types.rs`。

### 8.2 最大技术风险：平台签名

各平台签名模块（`sign.rs`）是发布失败的首要原因。推荐方案：

1. **优先使用 JS 引擎**：`boa_engine` 执行平台原始签名脚本，减少逆向维护成本
2. **备选外部签名服务**：将签名逻辑部署为独立微服务，方便热更新
3. **降级处理**：签名失败返回 `PLAT_004` 错误码，提示用户平台 API 可能变更

### 8.3 平台适配分批策略

严格按路线图分批，先易后难：

- **第一批**：微博（无签名）、头条（简单 token）、公众号（官方 API）
- **第二批**：B站（wbi 签名）、小红书（X-s/X-t）、抖音（X-Bogus）

### 8.4 sqlx 编译期检查

`sqlx::query!` / `sqlx::query_as!` 宏需要编译期数据库连接。两种方案：

- **方案 A**：设置 `DATABASE_URL` 环境变量指向 SQLite 文件
- **方案 B**：使用 offline 模式，先运行 `cargo sqlx prepare` 生成查询元数据，之后无需数据库即可编译
